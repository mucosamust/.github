# Contributing to MUST Computing Projects 🚀

Thank you for taking the time to contribute! This organization is built and maintained by students, for students. By participating, you are helping build a vibrant open-source culture at MUST.

To keep our repositories clean, structured, and production-ready, please follow these engineering guidelines.

---

## 🗺️ Git Workflow Overview

We use a standard Feature Branch workflow. Follow these steps for every contribution:

1. **Find an Issue:** Browse the repository issues. Comment on what you want to work on, or open a new issue describing a bug or feature.
2. **Fork / Branch:** Create a descriptive branch off of `main`.
3. **Commit often:** Keep your commits small, isolated, and well-named.
4. **Pull Request (PR):** Push your branch to GitHub and open a PR against the upstream `main` branch.
5. **Code Review:** A peer or maintainer will review your code, suggest changes if necessary, and merge it.

---

## 🌿 Branch Naming Conventions

Before you write code, ensure your branch name follows our community standard. Use lowercase characters and hyphens:

*   `feat/` — For brand new features (e.g., `feat/offline-sync`, `feat/auth-layout`)
*   `fix/` — For bug fixes (e.g., `fix/null-pointer-drc`, `fix/navbar-overflow`)
*   `docs/` — For documentation updates (e.g., `docs/update-readme`)
*   `refactor/` — For code cleanup or restructuring that doesn't change behavior

---

## 📝 Commit Message Guidelines

We enforce clean, descriptive commit messages. Avoid generic messages like `fixed bug` or `updates`. Instead, follow this structured format:

```text
- type(scope): short description in present tense, optional bullet point detailing what changed, why it was changed

```

Examples

---
```text
- feat(ui): add Compose-based dashboard layout for mobile clients

- fix(core): resolve race condition in distributed log sync

- docs(readme): fix broken community chat links
```

## 🚀 Pull Request Checklist

When opening a Pull Request, please ensure the following:

- [] One PR per feature: Do not bundle multiple unrelated fixes or features into a single PR.

- [] Sync with main: Ensure your branch is rebased or merged with the latest changes from main to avoid conflicts.

- [] Run local checks: Test your code locally before pushing (e.g., run compilers, linters, or hardware rule checks).

- [] Fill the template: Complete the PR description explaining what was changed and why.

## 🤝 Code of Conduct & Peer Review

**Be Constructive:** We are all learning. When reviewing a peer's PR, point out improvements politely and explain the reasoning.

**Be Responsive:** If a reviewer requests changes on your PR, address them promptly or start a discussion in the comments.

**Don't Discourage Anyone:** No question is too basic, and no bug is too small. Help your fellow peers grow.
